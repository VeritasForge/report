[주간 보고서 작성 지침]

## 목적
- Confluence 페이지와 JIRA 티켓을 읽고 대표에게 보고할 주간 업무 보고서를 작성합니다.

## 작성 원칙
- 단순히 JIRA 티켓을 나열하지 않고, 내용을 추상화하여 정리합니다.
- 경영진이 이해할 수 있도록 기술 용어는 최소화하고 비즈니스 관점에서 설명합니다.
- 각 항목은 "무엇을 했는지/하고 있는지"가 명확히 드러나도록 작성합니다.
- 불필요한 세부사항은 생략하고 핵심만 간결하게 전달합니다.

## 제품 및 Component 기반 그룹핑
- JIRA 티켓의 Details > Components 필드를 확인하여 작업 내용을 분류합니다.
- Component명의 prefix로 제품을 구분합니다:
  - `vc-` prefix: VC 제품
  - `er-` prefix: ER 제품
- 반드시 제품별(VC/ER)로 먼저 구분한 후, 각 제품 내에서 진행 상태별, Component별로 그룹핑합니다.
- 같은 Component에 해당하는 작업들은 하나의 그룹으로 묶어서 표현합니다.

## 보고서 형식
```
VC

[진행 완료]
- {Component명}
  - 완료된 작업 항목
  - 완료된 작업 항목

[진행 중]
- {Component명}
  - 진행 중인 작업 항목

[진행 대기]
- 특이사항 없음

ER

[진행 완료]
- {Component명}
  - 완료된 작업 항목

[진행 중]
- {Component명}
  - 진행 중인 작업 항목

[진행 대기]
- 특이사항 없음
```

## 작성 시 주의사항
- 제품(VC/ER)을 반드시 분리하여 작성합니다. 절대 섞이면 안 됩니다.
- 같은 Component의 작업은 반드시 하나로 묶어서 표현합니다.
- 버그 수정은 "어떤 문제를 해결했는지" 관점으로 작성합니다.
- 기능 개발은 "어떤 기능이 추가/개선되었는지" 관점으로 작성합니다.
- 티켓 번호나 기술적 구현 상세는 포함하지 않습니다.
